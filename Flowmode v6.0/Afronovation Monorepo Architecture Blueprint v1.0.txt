# Afronovation Monorepo Architecture Blueprint v1.0

## 1. Purpose

### Executive Summary

The Afronovation Monorepo Architecture Blueprint defines the canonical structure of the Afronovation codebase.

Its purpose is to ensure that:

* all products follow one system
* shared capabilities are reused
* governance is enforceable
* AI tools do not create structural drift
* new apps and services can be added cleanly
* platform assets scale across Bridge55, EmbassyOS, BridgeVault, BridgeAPI, Afronovation.com, and future products

This blueprint is the **physical realization** of FlowMode v6 and the Master Tech Stack Register.

### Operational Purpose

This blueprint defines:

* top-level folder zones
* module types
* ownership boundaries
* rules for shared vs app-specific code
* template expectations
* how overlays are added
* how IDEs should enforce structure

### System Rule

```yaml
monorepo_blueprint:
  required_for_all_modules: true
  prohibit_unapproved_top_level_folders: true
  require_template_based_generation: true
  require_module_registration: true
```

---

## 2. Architectural Principles

### Executive Summary

The monorepo is designed around five principles:

1. Shared where possible
2. Isolated where necessary
3. Governed by default
4. Template-driven from day one
5. Expandable without restructuring

### Operational Guidance

The monorepo must support:

* shared platform services
* multiple apps
* multiple domain overlays
* central design system assets
* central AI and orchestration assets
* central documentation and governance
* future OEM and enterprise client needs

### System Rule

```yaml
architecture_principles:
  prefer_shared_platform_assets: true
  prefer_modular_boundaries: true
  prohibit_duplicate_foundational_logic: true
  require_expandable_structure: true
  require_clear_ownership: true
```

Infrastructure-as-code, provider configs, environment definitions, deployment manifests.

Shared configuration schemas, policy configs, model configs, tenant configs, environment configs.

Setup, validation, migration, bootstrap, release, and repo operation scripts.

## 5. Zone-by-Zone Rules

## 5.1 `apps/` Rules

### Purpose

`apps/` contains user-facing and operator-facing products. Apps consume platform capabilities; they do not recreate them.

### Allowed

* app-specific routes and pages
* app-specific components
* app-specific workflows
* app-level config
* app-level docs
* app-level tests
* domain overlay bindings

### Not Allowed

* duplicated shared primitives
* direct provider integrations that belong in services
* duplicate auth logic
* duplicate design token systems
* duplicate payment logic
* duplicate feature flag infrastructure

### Recommended App Structure

```plaintext
apps/<app-name>/
```

## 8. Overlay Architecture Model

### Executive Summary

New app and domain overlays will keep being added. The repo must support that without restructuring.

### Overlay Types

* product/domain overlay
* regional overlay
* OEM/client overlay
* UI/theme overlay
* compliance/policy overlay

### Suggested Overlay Locations

* `docs/overlays/`
* `configs/tenants/`
* `packages/design-system/`
* service-level config/policy layers where needed

### Overlay Rules

* overlays may extend, not break, core behavior
* overlays must declare dependencies
* overlays must declare overrides
* overlays must be versioned if operationally significant

### System Rule

```yaml
overlay_architecture:
  support_incremental_overlay_growth: true
  require_overlay_definition: true
  prohibit_core_platform_forking: true
  require_override_registry: true
```

## 13. Boundary Rules: What Must Be Shared vs Local

### Must Be Shared

* design tokens
* shared UI primitives
* shared auth primitives
* shared types
* analytics base utilities
* feature flag infrastructure
* model routing policies
* skill and tool registries
* evaluation logic
* API mediation patterns

### Must Stay Local

* app-specific page composition
* app-specific workflow logic
* app-specific copy/content
* product-specific experiments
* contextual UI built on shared primitives

### Review Before Promoting to Shared

* business logic that may later generalize
* product-specific workflow helpers
* region-specific integrations
* experimental AI behaviors

### System Rule

```yaml
boundary_rules:
  require_shared_for_foundational_assets: true
  require_local_for_product_specific_logic: true
  require_review_before_promoting_to_shared: true
```
