# Afronovation Master Tech Stack Register v1.1

## 1. Purpose

### Executive Summary

The Afronovation Master Tech Stack Register is the authoritative record of approved technology capabilities, platform defaults, alternatives, scaling paths, and usage rules across the Afronovation ecosystem.

It is designed to prevent:

* stack drift
* duplicated architecture decisions
* unnecessary rewrites
* inconsistent implementation patterns
* avoidable vendor sprawl
* weak governance over future platform growth

This register is intentionally **capability-based**, not merely vendor-based.
Afronovation defines **the capability first**, then maps technologies to that capability.

### Operational Purpose

This register defines:

* capability categories
* default technologies
* approved alternatives
* enterprise conditional paths
* when to use each option
* when not to use each option
* owners and review cadence

### System Rule

```yaml
tech_stack_register:
  required_for_all_projects: true
  capability_based: true
  prohibit_ungoverned_stack_sprawl: true
  require_module_level_stack_mapping: true
  require_documented_exception_for_nonstandard_choices: true
```

---

## 2. Guiding Principles

### Executive Summary

Afronovation stack decisions are based on six principles:

1. Speed with structure
2. Enterprise-grade scalability
3. Sovereign-grade trust
4. Multi-tenant readiness
5. Pragmatic affordability
6. Vendor flexibility without architectural chaos

### Operational Guidance

A technology should be preferred when it:

* solves a repeatable platform problem
* reduces delivery time without weakening governance
* supports multi-tenant or enterprise patterns
* fits the FlowMode v6 operating model
* has a clear scaling path
* can be abstracted or swapped with minimal platform disruption

### System Rule

```yaml
stack_principles:
  prefer_capabilities_over_vendor_lock_in: true
  prefer_reuse_over_novelty: true
  prefer_governed_platforms: true
  prefer_enterprise_scale_paths: true
  prefer_ai_native_compatibility: true
  require_reviewable_tradeoffs: true
```

---

## 3. Stack Classification Model

### Executive Summary

Every stack decision must be classified so teams know whether a technology is default, alternative, conditional, or experimental.

### Classification Levels

* **Tier A — Core Approved Default**
  Mandatory unless an approved exception exists.

* **Tier B — Approved Alternative**
  Allowed when there is a valid team, delivery, or module reason.

* **Tier C — Enterprise Conditional**
  Used when compliance, client, scale, or geographic requirements demand it.

* **Tier D — Experimental**
  Allowed only in pilots, sandboxes, or reviewed evaluation contexts.

### System Rule

```yaml
stack_classification:
  tiers:
    - core_default
    - approved_alternative
    - enterprise_conditional
    - experimental
  require_tier_assignment: true
```

## 5.3 IDE-Aligned Model Guidance

### Claude-Centered Work

* primary: reasoning-focused model
* secondary: lower-cost transformation model
* optional: higher-stakes reasoning tier
* fallback: routed secondary provider

### Cursor-Centered Work

* primary: strong coding model
* secondary: reasoning model
* tertiary: long-context model
* low-cost coder for repetitive tasks

### Antigravity-Centered Work

* primary: orchestration / multimodal model
* secondary: faster low-latency model
* fallback: reasoning model
* coding fallback where needed

### System Rule

```yaml
ide_model_guidance:
  claude:
    prefer:
      - reasoning_primary
      - lightweight_transform
      - high_stakes_reasoning
      - routed_fallback
  cursor:
    prefer:
      - coding_primary
      - reasoning_secondary
      - long_context_secondary
      - low_cost_coder
  antigravity:
    prefer:
      - orchestration_primary
      - fast_secondary
      - reasoning_fallback
      - coding_fallback
```

## 6.4 Payments Capability

### Executive Summary

BridgePayments is not a single payment integration. It is a **payment orchestration and normalization layer** that lets Afronovation apps support multiple processors, regional payment methods, tenant-specific enablement, wallet/credit patterns, and future growth without rewriting each app.

### Capability Table

| Capability                           | Purpose                                      | Default Technology                       | Approved Alternatives             | Enterprise Conditional                      | Use When                                        | Do Not Use When                                        | Owner                              | Review Cadence |
| ------------------------------------ | -------------------------------------------- | ---------------------------------------- | --------------------------------- | ------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------ | ---------------------------------- | -------------- |
| Payment orchestration                | Unified payment logic and routing            | BridgePayments                           | None preferred                    | Client-dedicated payment routing layer      | Any app requiring payments or monetization      | Apps call payment providers directly                   | Payments Platform                  | Every 6 months |
| Global/general processor             | Broad processor support                      | Stripe adapter                           | Additional processor adapters     | Client requires different primary processor | Standard checkout, subscriptions, cards         | Regional fit or method mix requires localization       | Payments Platform                  | Every 6 months |
| Africa-focused processor             | Regional payment support                     | Paystack adapter                         | Flutterwave adapter               | Market-specific provider expansion          | West Africa and regional payment fit            | Geography or method coverage is insufficient           | Payments Platform                  | Every 6 months |
| Multi-method Africa/global processor | Additional channels and regional flexibility | Flutterwave adapter                      | Paystack adapter                  | Future provider additions                   | Cards, transfers, wallets, mixed method support | Method support or commercial alignment is insufficient | Payments Platform                  | Every 6 months |
| Mobile money adapter                 | Mobile-money payment support                 | M-Pesa adapter                           | Other mobile money adapters later | Market-specific deployments                 | Regional mobile money acceptance is required    | Market does not need it                                | Payments Platform                  | Every 6 months |
| Wallet and credits                   | Stored value and prepaid usage               | BridgePayments wallet layer              | None preferred                    | Product-specific wallet variant             | Usage credits, OEM billing, internal balances   | Product does not need stored value                     | Payments Platform                  | Every 6 months |
| Billing model enforcement            | Subscription, usage, tenant, hybrid billing  | Product-configured BridgePayments policy | None preferred                    | Client-specific contract billing logic      | Monetized products and OEM packaging            | Billing is handled ad hoc outside governance           | Payments Platform + Commercial Ops | Every 6 months |

### BridgePayments Platform Layers

**Layer 1 — Core Payment Domain**

* payments
* refunds
* payouts
* subscriptions
* invoices
* wallet / credits
* reconciliation

**Layer 2 — Provider Adapters**

* Stripe
* Paystack
* Flutterwave
* M-Pesa
* future aggregators or regional processors

**Layer 3 — Policy and Routing**

* tenant-enabled providers
* method resolution
* region / currency rules
* fee logic
* fallback logic
* billing model hooks
* risk controls
* audit and reconciliation hooks

### System Rule

```yaml
payments_stack:
  gateway_module: bridgepayments
  support_multi_processor_orchestration: true
  support_wallet: true
  support_credit_model: true
  support_provider_routing: true
  support_tenant_method_enablement: true
  require_product_level_billing_definition: true
  prohibit_direct_provider_calls_from_apps: true
```

# 15. Exception and Change Control

### Executive Summary

This register is designed to reduce rewrites by staying capability-first. Technologies may evolve, but changes must be governed.

### Approved Change Process

A new technology may be introduced only if it includes:

* capability gap being solved
* rationale
* cost impact
* security impact
* operational impact
* migration implications
* owner
* review decision
