Below is the **Afronovation Skill Manifest v1**.

This document turns FlowMode from an operating model into an executable system by defining the first governed skill inventory, how each skill behaves, when it is used, who owns it, what it produces, and how it is approved.

It is designed to be:

* executive-readable
* engineering-actionable
* AI/IDE-enforceable

# Afronovation Skill Manifest v1

## 1. Purpose

### Executive Summary

The Afronovation Skill Manifest defines the governed AI skills that power execution across Afronovation’s platform, apps, services, reviews, QA, design, security, and release workflows.

A skill is not just a prompt.
A skill is a **versioned, owned, testable, policy-bound execution capability**.

This manifest exists to ensure that AI work is:

* consistent
* modular
* governed
* reusable
* measurable
* scalable across current and future products

### Operational Purpose

This manifest defines:

* skill categories
* first-wave production skills
* ownership and risk tiering
* inputs and outputs
* required artifacts
* approval rules
* execution expectations
* future extension model for app/domain overlays

### System Rule

```yaml
skill_manifest:
  required_for_ai_execution: true
  prohibit_unregistered_production_skills: true
  require_owner_and_risk_tier: true
  require_evaluation_definition: true
```

## 2. Skill Design Principles

### Executive Summary

Afronovation skills must be designed as governed platform assets, not informal one-off instructions.

### Core Principles

1. **One skill = one clear execution role**
2. **Every skill must produce a useful artifact or outcome**
3. **Skills must be versioned and owned**
4. **Skills must align with FlowMode**
5. **High-risk skills require stronger approval**
6. **Skills should be reusable across products where possible**
7. **Domain overlays extend the system without breaking the core**

### Operational Guidance

A skill should exist when:

* a workflow repeats often
* consistency matters
* risk exists if execution is ad hoc
* evidence or artifacts are needed
* AI behavior must be constrained and reviewable

A skill should not exist when:

* the behavior is too vague
* the workflow is not repeatable
* the task is better handled by general reasoning inside a governed parent skill

### System Rule

```yaml
skill_principles:
  require_clear_execution_role: true
  require_artifact_or_decision_output: true
  require_governance_alignment: true
  require_owner: true
  require_versioning: true
```

## 5. Skill Lifecycle

### Executive Summary

Skills must move through a controlled lifecycle.

### Lifecycle Stages

* **draft** — being created or refined
* **review** — under governance and quality review
* **approved** — allowed for limited or controlled use
* **deployed** — available for standard use
* **deprecated** — no longer recommended; migration path required

### Operational Guidance

A skill may not be used in production if it is:

* unregistered
* unowned
* unevaluated
* missing risk tier
* missing approval policy

### System Rule

```yaml
skill_lifecycle:
  stages:
    - draft
    - review
    - approved
    - deployed
    - deprecated
  require_review_before_approval: true
  require_evaluation_before_deploy: true
  require_migration_note_for_deprecation: true
```

# 7. Domain Overlay Skill Framework

## 7.1 Executive Summary

Domain skills should not be hardcoded into the core manifest forever. The manifest must support incremental expansion as Afronovation launches new apps.

That means domain skills are treated as **overlay packs**.

## 7.2 Current Planned Overlay Families

* Bridge55
* EmbassyOS
* BridgeVault
* BridgeAPI
* BridgeAir
* BridgeStay
* BridgeMobility
* AfCON Hub
* future app/domain overlays

## 7.3 Overlay Skill Rule

Each overlay skill must:

* reference the base manifest
* declare domain dependencies
* declare special rules or constraints
* declare what core skills it depends on
* avoid duplicating shared logic already handled by core skills

# 10. Risk Tiering and Approval Matrix

## 10.1 Risk Tiers

| Risk Tier | Meaning                                                       | Approval Policy |
| --------- | ------------------------------------------------------------- | --------------- |
| Low       | Low-impact, non-sensitive, reversible                         | auto            |
| Medium    | Important but bounded impact                                  | reviewed        |
| High      | Material platform, product, trust, or customer impact         | senior_review   |
| Critical  | Security, data, auth, destructive, or highly sensitive impact | multi_approval  |

## 10.2 Approval Expectations

### Auto

May proceed if registered, valid, and within bounds.

### Reviewed

Requires standard review by the relevant owner domain.

### Senior Review

Requires elevated review and stronger artifact completeness.

### Multi Approval

Requires multiple stakeholders or roles before production use.

### System Rule

```yaml
approval_matrix:
  low: auto
  medium: reviewed
  high: senior_review
  critical: multi_approval
```

# 13. Skill Execution Rules

### Executive Summary

A skill should be invoked only when its role matches the task and its governance path is satisfied.

### Execution Rules

* do not invoke a skill outside its purpose
* do not skip required artifacts
* do not bypass higher-risk approval paths
* do not duplicate an existing registered skill unnecessarily
* use parent/core skills before creating overlay variants

### System Rule

```yaml
skill_execution_rules:
  require_role_match: true
  require_artifacts: true
  prohibit_approval_bypass: true
  prohibit_duplicate_skill_creation_without_review: true
```

# 15. Future Expansion Model

### Executive Summary

Afronovation will continue adding new apps and domain overlays. This manifest is designed for growth without requiring structural reinvention.

### Expansion Rules

When a new app is added:

1. evaluate whether core skills already cover most of the need
2. create overlay candidates only for domain-specific logic
3. register new overlay skills with dependencies
4. avoid cloning core skills unless governance-approved
5. update overlay docs and module registry

### System Rule

```yaml
future_skill_expansion:
  support_incremental_overlay_growth: true
  require_core_first_reuse_check: true
  require_overlay_registration: true
```
